# eBPF file-open tracer ‚Äî Phase 4 POC (exploratory, NOT integrated)

This directory is a **scoping proof-of-concept** for Phase 4 (eBPF monitoring).
It is deliberately isolated from the production goboxd code:

- it is its **own Go module** (`module ebpf-poc`), so the `cilium/ebpf`
  dependency never enters the root `github.com/nym01/goboxd` module graph;
- it touches **none** of `internal/runner`, `internal/api`, `nsjail.go`,
  `seccomp.policy`, the CLI, MCP, or the frontend.

The goal was narrow: **prove we can trace every file-open a short-lived,
PID-filtered sandboxed process makes, from outside the sandbox, in this
WSL2/Docker environment.** It does. See [FINDINGS.md](FINDINGS.md) for the full
write-up (kernel/cap details, the PID-namespace gotcha, and the attach-race
assessment that matters for the eventual integration design).

## What it does

Attaches eBPF programs to the `sys_enter_openat` and `sys_enter_openat2`
tracepoints, filters **in-kernel** to a single target process, and streams
`(timestamp, pid, syscall, filename)` to stdout over a ring buffer.

Because the monitor and the kernel sit in different PID namespaces here (and
nsjail nests the sandboxed child one level deeper still), the filter does **not**
compare the raw pid. It identifies the target's PID *namespace* by the
`(dev, inode)` of `/proc/<pid>/ns/pid` and compares the pid *within that
namespace* via `bpf_get_ns_current_pid_tgid()`. See `trace.bpf.c`.

## Files

| File | Purpose |
|------|---------|
| `trace.bpf.c`      | the eBPF program (tracepoints + ns-aware PID filter + ringbuf) |
| `main.go`          | the user-space loader/reader (cilium/ebpf) |
| `go.mod`           | isolated module pinning `github.com/cilium/ebpf` |
| `build-and-run.sh` | installs the toolchain, builds, and runs a self-test |
| `.gitignore`       | excludes generated artifacts (`vmlinux.h`, `*.o`, bindings, binary) |

Generated by the build (git-ignored): `vmlinux.h`, `trace_bpfel.go` /
`trace_bpfeb.go` (bpf2go bindings), `trace_bpfel.o` / `trace_bpfeb.o`, and the
`ebpf-poc` binary.

## Build & run

Requires a **privileged** Linux container with the host kernel's BTF at
`/sys/kernel/btf/vmlinux` (true for Docker Desktop on WSL2). From the repo root:

```bash
# MSYS_NO_PATHCONV=1 stops Git-Bash mangling the container paths on Windows.
MSYS_NO_PATHCONV=1 docker run --rm --privileged --pid=host \
  -v "D:/PROJECTS/Tracebox/experiments/ebpf-poc:/poc" -w /poc \
  golang:1.25 bash /poc/build-and-run.sh
```

`--pid=host` lets the throwaway test process and the tracer share a PID
namespace so the self-test can resolve the target. `--privileged` provides
CAP_BPF/CAP_PERFMON and the ability to mount tracefs.

### Toolchain (for the eventual Dockerfile)

Built on the `golang:1.25` image (Go 1.25 ‚Äî Debian's `golang` 1.19 is too old
for cilium/ebpf) plus, via apt:

- `clang llvm`     ‚Äî compile `trace.bpf.c` ‚Üí BPF ELF object
- `libbpf-dev`     ‚Äî `<bpf/bpf_helpers.h>` etc.
- `linux-libc-dev` ‚Äî kernel uapi headers
- `bpftool`        ‚Äî dump kernel BTF to `vmlinux.h`

CGO is **not** required: bpf2go embeds the compiled `.o` into generated Go.

## Manual usage

```bash
sudo ./ebpf-poc <pid>   # trace one process (ns-aware)
sudo ./ebpf-poc 0       # firehose: trace openat/openat2 for every process
```

## Regenerating bindings

```bash
go generate ./...   # re-runs bpf2go (needs clang + vmlinux.h present)
```

## Note for editors on Windows

The Go LSP will flag `syscall.Stat_t`, `cilium/ebpf` imports, and the
`trace*` generated identifiers as unresolved on Windows ‚Äî expected: this is
Linux-only code whose bindings are generated inside the Linux build container.
It compiles cleanly there (`go build` prints no errors).
