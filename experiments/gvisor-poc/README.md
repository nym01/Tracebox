# gVisor (runsc) feasibility — Phase 7 POC (exploratory, NOT integrated)

This directory is a **scoping proof-of-concept** for Phase 7 (pluggable isolation),
asking whether **gVisor's `runsc`** could run as an alternative to nsjail — and in
particular whether it fixes **security-audit Finding F** (the `/proc` host-info
leak in `docs/security-audit-findings.md`). It mirrors the `experiments/ebpf-poc/`
pattern: isolated, empirical, generated artifacts git-ignored.

It touches **none** of the production code — `internal/runner/`, `nsjail.go`,
`buildNsjailArgs`, `seccomp.policy`, the CLI, MCP, or the frontend. It only
downloads `runsc`, builds a throwaway OCI bundle, and runs read-only probes.

**Verdict (short): feasible, and it fixes Finding F.** gVisor runs in this
WSL2 + Docker Desktop environment via the **systrap** platform (no KVM needed),
runs py3 under a minimal OCI bundle, and presents a **synthesised `/proc`** that
no longer leaks the host kernel version, CPU model, live loadavg, or (with limits
applied) host RAM/CPU count. Resource limits are honoured via the OCI spec
(memory + CPU verified end-to-end; the `pids` controller trips a startup failure
*in this specific nested environment*). The existing eBPF tracer would **not**
work as-is — gVisor handles guest syscalls in its userspace "sentry," so they
never hit the host tracepoints the tracer attaches to; gVisor's own tracing
(`runsc --strace` / trace points) is the replacement. Full write-up, with the
actual commands and output, in [FINDINGS.md](FINDINGS.md).

## Build & run

Requires a **privileged** Linux container started with **`--cgroupns=host`** —
the *same* deployment shape the nsjail sandbox already uses (see
`docker-compose.yml`). The `python:3-slim` base supplies python3 + its shared
libraries to populate the OCI rootfs. From the repo root, on Windows/Git-Bash:

```bash
# MSYS_NO_PATHCONV=1 stops Git-Bash mangling the container paths on Windows.
MSYS_NO_PATHCONV=1 docker run --rm --privileged --cgroupns=host \
  -v "D:/PROJECTS/Tracebox/experiments/gvisor-poc:/poc" -w /poc \
  python:3-slim bash /poc/build-and-run.sh
```

`--cgroupns=host` lets runsc create its per-sandbox cgroup under the host v2
hierarchy (without it, `runsc` hits `cgroup.subtree_control: device or resource
busy` and you must fall back to `--ignore-cgroups`, which disables limits). To
keep the 250 MB rootfs off the Windows filesystem you can instead pipe the script
in (`... -i python:3-slim bash -s < build-and-run.sh`), which runs entirely on the
container's own fs.

> **Note:** sandbox startup can fail with `cannot read client sync file: ... EOF`
> in this nested WSL2/Docker environment. Root cause (FINDINGS.md §4/§6): the
> gVisor sentry forks helper tasks at startup and hits `EAGAIN` (`newosproc`) when
> the host is short on thread/pid headroom — so it worsens after many sandboxes
> have been spun up in one session, and a tight `pids.max` triggers it
> deterministically. Expected to clear on a real Linux host; until then some probes
> may need a rerun (or a fresh container).
