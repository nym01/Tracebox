# Phase 7 gVisor POC ‚Äî Findings

**Scope:** determine whether gVisor (`runsc`) can run as an alternative to nsjail
for this project, what it would take, and specifically whether it fixes
**security-audit Finding F** (the `/proc` host-info leak in
`docs/security-audit-findings.md`). Exploratory only ‚Äî nothing here is wired into
the goboxd API/runner/CLI/MCP, and no production file was modified.

**Verdict: feasible in this environment, and it fixes Finding F.** `runsc` runs
under the **systrap** platform with **no KVM** (which is absent here), runs py3
under a minimal OCI bundle, and presents a synthesised `/proc` that closes the
Finding-F leaks. Memory and CPU limits are honoured via the OCI spec (a 3 GB bomb
under a 512 MB limit was OOM-killed, exit 137); a tight `pids` limit ‚Äî and, under
host thread pressure, sandbox startup generally ‚Äî trips a sentry-startup failure
that traces to one root cause (`newosproc`/`EAGAIN`: the sentry can't fork its own
helper tasks), detailed in ¬ß4/¬ß6. The existing eBPF tracer would **not** see a
gVisor-run process ‚Äî that needs a different data source (gVisor's own tracing). A
v1 `GvisorRunner` (py3-only, basic limits, Finding F fixed) is a **few-days** task,
not blocked.

All versions/commands below were run live. Environment: **runsc
`release-20260608.0`** in a `--privileged --cgroupns=host` container on **Docker
Desktop 4.77 / WSL2 kernel `6.6.114.1-microsoft-standard-WSL2`**.

## 1. Installation & invocation model

### Can runsc run inside the existing privileged container? ‚Äî **Yes.**

`runsc` is a single static binary; no daemon, no kernel module. Downloaded from
gVisor's release bucket and run directly **inside** the same kind of container the
project already uses for nsjail:

```
$ runsc --version
runsc version release-20260608.0
spec: 1.2.1
```

It does **not** require Docker's `--runtime=runsc` / Docker-in-Docker model. The
project could keep its current "one privileged container, sandbox launched as a
child process" shape and simply launch `runsc run <bundle>` where it launches
`nsjail ‚Ä¶` today. (Wiring runsc as a *Docker runtime* is the other deployment
option, but it is not necessary and would be a bigger change ‚Äî the
launch-a-child-process model maps cleanly onto the existing `Runner` interface.)

### Environment-specific blockers (the WSL2 / no-KVM question)

| Probe | Result |
|-------|--------|
| `/dev/kvm` inside privileged container | **ABSENT** (`ls: cannot access '/dev/kvm'`) |
| `vmx` in `/proc/cpuinfo` | present (but no `/dev/kvm` node is exposed by Docker Desktop) |
| gVisor **KVM** platform | **unavailable** (needs `/dev/kvm`) |
| gVisor **systrap** platform (default) | **works** ‚Äî no KVM, no special device |

This is the gVisor analogue of the eBPF POC's WSL2 kernel notes. **The KVM
platform is a non-starter here** (Docker Desktop does not pass `/dev/kvm` into
containers). gVisor's default **systrap** platform ‚Äî which traps guest syscalls
with `seccomp` + signals entirely in userspace ‚Äî needs no KVM and ran fine:

```
$ runsc --platform=systrap --network=none --ignore-cgroups do echo HELLO_FROM_GVISOR
HELLO_FROM_GVISOR
```

So gVisor is **not** blocked by the WSL2/no-KVM environment. (The trade-off:
systrap is slower than the KVM platform for syscall-heavy workloads. On a real
Linux host with `/dev/kvm`, the faster KVM platform would also be available ‚Äî a
performance upside, not a correctness blocker.)

### One cgroup gotcha in the invocation model

`runsc do` with the container's *default* cgroup namespace fails:

```
creating container: cannot set up cgroup for root: configuring cgroup:
write /sys/fs/cgroup/cgroup.subtree_control: device or resource busy
```

Two ways past it, exactly mirroring how nsjail is deployed:

- `--ignore-cgroups` ‚Äî runsc skips cgroup setup (fine for a smoke test, but
  **disables resource limits**, see ¬ß4); or
- run the container with **`--cgroupns=host`** ‚Äî the *same flag the nsjail
  sandbox already requires* (see `docker-compose.yml`). runsc then creates its
  per-sandbox cgroup under the host v2 hierarchy and limits work. This is the
  realistic deployment shape and is what ¬ß3/¬ß4 use.

## 2. Basic feasibility test ‚Äî py3 under gVisor

A minimal OCI bundle ran python3 under systrap isolation:

```
PY 3.14.6 under gVisor
```

**What the OCI bundle needs:**

- **`config.json`** ‚Äî generated by `runsc spec` (a standard OCI runtime spec),
  then `.process.args` pointed at `["python3", "/hello.py"]`. The default spec is
  ~200 lines; only a handful of fields matter for us (args, env, the `linux.resources`
  limits in ¬ß4, `cgroupsPath`).
- **A root filesystem** ‚Äî gVisor runs the container against a real rootfs
  directory (`config.json` ‚Üí `root.path`, default `rootfs/`). This is the main
  structural difference from nsjail: nsjail synthesises a minimal mount namespace
  by bind-mounting individual host files (the `resolve*Mounts` logic in
  `nsjail.go`), whereas gVisor wants a populated rootfs tree.

  **How minimal can the py3 rootfs be?** For this POC it was populated by `tar`ing
  the `python:3-slim` container's own `/bin /lib /lib64 /usr /etc /sbin`
  (**~250 MB**) ‚Äî quick but not minimal. A purpose-built py3 rootfs (python3 + its
  `ldd` closure + the stdlib, the same set `resolvePy3Mounts` already computes)
  would be a small fraction of that. Either way, gVisor's "give it a rootfs"
  model is a **different** filesystem story than nsjail's per-file bind mounts and
  is the single biggest piece of new work for an integration (see ¬ß6).

## 3. Finding F check ‚Äî the payoff question

Read `/proc/{cpuinfo,meminfo,version,loadavg}` (escape test 20 / Finding F) from
inside the sandbox. **Host = what the nsjail sandbox leaks today; gVisor = what
the sentry synthesises.**

| `/proc` source | nsjail (host leak today) | **gVisor (systrap)** | Finding F? |
|----------------|--------------------------|----------------------|------------|
| `version` | `6.6.114.1-microsoft-standard-WSL2 ‚Ä¶ gcc 13.2.0` | **`4.19.0-gvisor`** (synthetic) | **FIXED** |
| `cpuinfo` model name | `11th Gen Intel(R) Core(TM) i5-1135G7` | **`unknown`** | **FIXED** |
| `loadavg` | `0.26 0.14 0.05 1/292 2683` (live host) | **`0.00 0.00 0.00 0/0 0`** | **FIXED** (side channel closed) |
| `meminfo` MemTotal | `7977068 kB` (host 8 GB) | **`262144 kB`** = the 256 MB sandbox limit¬π | **FIXED**¬π |
| cpu count (`nproc`) | `8` (host) | **`2`** = derived from the CPU quota¬π | **FIXED**¬π |

¬π **only when a limit is applied** (`--cgroupns=host`, OCI `memory.limit` /
`cpu.quota` set ‚Äî see ¬ß4). With `--ignore-cgroups` and no limit, gVisor reports
the host's `MemTotal` (7977068 kB) and host cpu count (8) ‚Äî so the meminfo/cpu
parts of Finding F are fixed **via the resource limits**, while version / cpuinfo
model / loadavg are synthesised unconditionally.

**This is the concrete answer: switching to gVisor fixes Finding F.** The whole
*class* the audit described ‚Äî host kernel version, CPU model and the live
co-tenant side channel (`loadavg`/`MemFree` fluctuating with other tenants) ‚Äî is
closed by the sentry presenting its own `/proc`, with **no per-file masking** and
none of the JVM/Node breakage risk that made masking unattractive under nsjail
(the audit's stated reason F was left as a known limitation; it even names gVisor
as the clean fix).

**Two honest caveats:**

- `/proc/cpuinfo` still exposes `cpu family : 6`, `model : 140` and the full CPU
  **flags** list. The human-readable model string is gone, but family/model 6/140
  still identifies the microarchitecture (Tiger Lake). So cpuinfo is *substantially*
  de-fingerprinted, not perfectly anonymised.
- The meminfo/cpu-count fix is **contingent on running with a resource limit**
  (which the project does ‚Äî every run carries `memory.max`/`cpu.max` today), not on
  gVisor alone.

## 4. Resource limits ‚Äî rough check

Limits are expressed in the **standard OCI spec** (`linux.resources.{memory,cpu,pids}`)
‚Äî there is no separate gVisor config format to learn; the same `config.json` that
defines the container carries them. runsc reads them (confirmed in the debug log:
`created cgroup mounts for controllers [cpu cpuacct cpuset ‚Ä¶ memory pids]`).

| Limit | nsjail flag today | gVisor (OCI spec) | Result in this env |
|-------|-------------------|-------------------|--------------------|
| Memory | `--cgroup_mem_max` | `memory.limit` (bytes) | **enforced** ‚Äî see below |
| CPU | `--cgroup_cpu_ms_per_sec` | `cpu.quota` / `cpu.period` | **honoured** ‚Äî sizes guest cpu count (`cpus: 2` for 0.5-core quota); throttles |
| PIDs | `--cgroup_pids_max` | `pids.limit` | **trips a sandbox-start failure** in this env (see below) |

**Memory ‚Äî enforced (hard OOM-kill).** A guest allocating 3 GB under a 512 MB
limit was SIGKILLed:

```
### memory bomb: limit=swap=512MB, guest tries 3GB ###
exit=137          # 137 = 128+SIGKILL = OOM-killed == enforced
```

Two gVisor-specific wrinkles worth recording (both manageable, neither fatal):
- The limit covers the **sentry process + guest combined**, and the sentry has its
  own overhead: a 256 MB limit was too small for the sandbox to even *start*
  (`waiting for sandbox to start: EOF`), while 512 MB started and OOM-killed the
  bomb (exit 137) when the host had thread/memory headroom. (Startup reliability is
  itself pressure-dependent ‚Äî see ¬ß6.) An integration must size the limit =
  (language `memory_kb`) **+ sentry headroom**, unlike nsjail where the limit
  applies to the guest alone.
- WSL2's cgroup v2 has **no swap controller** (`memory.swap.max` absent at the cg
  root; controllers are `cpuset cpu io memory hugetlb pids rdma`). gVisor still
  enforces the cap because the sentry also bounds *guest* RAM internally
  (`loader.go: Setting total memory to ‚Ä¶`), which is what drives the OOM. nsjail,
  by contrast, needed `--cgroup_mem_swap_max 0` to make its limit hard.

**CPU ‚Äî honoured.** `cpu.quota=50000 / period=100000` (0.5 core) made the guest
report `cpus: 2` (gVisor's own rounding) instead of the host's 8, and the CPU
controller is set on the sandbox cgroup. Bandwidth throttling is the same model as
nsjail's `cpu.max`.

**PIDs ‚Äî environment blocker here, and it's the same root cause as the
startup flakiness (¬ß6).** Adding `pids.limit` to the spec made the sandbox fail to
start (`cannot read client sync file: ‚Ä¶EOF`), **deterministically** (0/5 with pids
set; the identical config without `pids` starts fine). Capturing the runsc
`--debug` log shows the real cause:

```
FATAL ERROR: error executing umounter: fork/exec /proc/self/exe: resource temporarily unavailable
runtime: failed to create new OS thread (have 5 already; errno=11)
fatal error: newosproc
```

`errno=11` is **EAGAIN** on thread/process creation. **gVisor's sentry is itself
multi-threaded and forks a helper (`umounter`) at startup**, so it needs several
tasks of its own *before* any guest code runs. A tight `pids.max` (or a
thread-exhausted host ‚Äî see ¬ß6) denies that and the sentry dies during
bring-up ‚Üí the "client sync file EOF" the parent reports. So the OCI `pids`
resource *is* supported by runsc in general; the failure is that **the limit must
budget for the sentry's own tasks, not just the guest's** ‚Äî the analogue of the
memory wrinkle above (limit covers sentry + guest, not guest alone). An nsjail
`pids.max` of, say, 64 applies to the guest; the gVisor equivalent must add the
sentry's task overhead on top. Worth a clean retest on a real Linux host with a
generous `pids.max`.

**Bottom line for ¬ß4:** memory + CPU limits are *"supported, just translate the
existing per-language `RunSpec` numbers into the OCI `linux.resources` block ‚Äî
with sentry headroom added"*; the PIDs limit is the same story (*"supported, but
the cap must include the sentry's own tasks; tripped startup here and needs a
real-Linux retest with a larger budget"*).

## 5. eBPF tracer compatibility ‚Äî assessment (not implemented)

**The existing `internal/tracer/` would see essentially nothing useful for a
gVisor-run process, and would need a fundamentally different data source.**

How the current tracer works (from `internal/tracer/trace.bpf.c`): it attaches
eBPF programs to **host kernel** syscall-entry tracepoints
(`sys_enter_openat`/`openat2`, `execve`/`execveat`, `connect`) and filters by
`bpf_get_current_cgroup_id()` against the run's cgroup. This assumes the sandboxed
process issues these syscalls **directly to the host kernel** ‚Äî which is true
under nsjail (the child runs on the host kernel, just namespaced/seccomp-filtered).

Under gVisor that assumption breaks. With the **systrap** platform, the guest's
syscalls are trapped (via seccomp) and serviced by the **sentry** ‚Äî a userspace
process implementing its own kernel (VFS, netstack, ‚Ä¶). The guest's `openat` is
**handled inside the sentry**; it does **not** appear as a host `sys_enter_openat`.
The only host syscalls are the ones the *sentry itself* makes (and file I/O is
typically brokered to a separate **gofer** process over a unix socket). So a
host-tracepoint tracer keyed on the guest would:

- miss the guest's `openat`/`execve`/`connect` entirely (they never reach the host
  tracepoints as such), and
- at best see the sentry's/gofer's own, differently-shaped host activity ‚Äî not a
  faithful per-run file/exec/connect audit.

The cgroup-id correlation the tracer relies on is *partly* still valid (the sentry
runs in the run's cgroup), but the **events themselves are in the wrong layer**.

**The replacement is gVisor's own tracing.** Demonstrated live with `runsc
--strace`, which captured the guest's opens cleanly from the sentry:

```
python3 E openat(AT_FDCWD /, ‚Ä¶ /etc/hostname, O_RDONLY|O_CLOEXEC, 0o0)
python3 X openat(AT_FDCWD /, ‚Ä¶ /etc/hostname, ‚Ä¶) = 3 (34.278¬µs)
```

That is exactly the `(syscall, path, result)` data the eBPF tracer produces ‚Äî
same information, **different source**. For a real integration gVisor offers a
structured path beyond `--strace`: its **Runtime Monitoring / trace points** API
(seccomp-notify-style trace sessions emitting structured syscall events, the
mechanism Falco-style tools consume). So Phase 4's tracer would be **replaced**,
not reused, under gVisor ‚Äî the *concept* (per-run syscall audit correlated to a
run) carries over; the *implementation* (host eBPF tracepoints) does not.

This also means the seccomp/`bpf(2)` interplay changes: the eBPF monitor runs at
container level *outside* the jail today, which is orthogonal to gVisor ‚Äî but the
in-sandbox `bpf(2)` deny in `seccomp.policy` is moot under gVisor anyway, since the
guest never touches the host kernel's bpf.

## 6. Overall assessment

### Effort estimate: a basic `GvisorRunner` is a **"few days"** task ‚Äî not blocked.

The hard questions are all answered "yes": runsc runs here (no KVM needed), py3
runs under it, Finding F is fixed, and memory/CPU limits work. Nothing about the
WSL2/Docker environment fundamentally blocks it. It is *more* than a few hours
because of the rootfs model and a couple of environment rough edges, but it is
squarely a few-days POC, not a multi-week effort and not "blocked."

**Not blocked by the environment.** The only hard environmental constraint ‚Äî
no `/dev/kvm` ‚Äî is sidestepped by the systrap platform, which works. The cgroup
items are *deployment* details the project already solves for nsjail
(`--cgroupns=host`).

The two "rough edges" turned out to be **one root cause** (see ¬ß4): the sandbox
sometimes fails to start with `cannot read client sync file: ‚Ä¶EOF`, and the
`--debug` log shows `fork/exec /proc/self/exe: resource temporarily unavailable` /
`failed to create new OS thread (‚Ä¶ errno=11)` / `fatal error: newosproc`. The
**sentry is multi-threaded and forks a `umounter` helper at startup**, so it needs
host thread/process headroom *before* the guest runs. It reproduces (a) when a
tight `pids.max` is set, and (b) intermittently under host thread pressure ‚Äî
during this POC it went from "starts fine, OOM-kills the bomb (137)" early on to
"0/12 starts" after dozens of throwaway sandboxes had been spun up in the same
WSL2 session, i.e. the host's task budget was the variable. On a real Linux host
(more pid/thread headroom, and a generous `pids.max` that budgets for the sentry's
own tasks) this is expected to clear; it also unlocks the faster KVM platform.
Either way it is a startup-provisioning detail, not an isolation gap ‚Äî worth a
clean retest on real Linux before relying on the limits.

### Rough scope for a v1 `GvisorRunner` (py3 only, Finding F fixed)

In scope (the few-days POC):

1. **A `GvisorRunner` implementing the existing `runner.Runner` interface** ‚Äî same
   `RunSpec`/`RunResult` shape as `NsjailRunner`, so callers don't change (the
   interface is explicitly designed for this swap). It would, per run: build an OCI
   bundle in the per-request work dir, write `config.json` (args = the py3 command,
   `cwd`, env `PATH`), and `runsc --platform=systrap --network=none run`.
2. **A minimal py3 rootfs**, built once at startup ‚Äî reuse `resolvePy3Mounts`'s
   existing `ldd`/stdlib discovery to assemble a small rootfs (or a prebuilt layer)
   instead of nsjail's per-file bind mounts. This is the main new component.
3. **Translate the `RunSpec` limits** into `linux.resources`: `MemoryKB` ‚Üí
   `memory.limit` **+ sentry headroom**, `CPUMsPerSec` ‚Üí `cpu.quota/period`. Defer
   `MaxProcesses`/`pids` until the startup-failure quirk is chased on real Linux.
4. **Map results**: exit 137 ‚Üí `memory_exceeded` (as `oomKilled` does today); wall
   limit via the OCI process/`--time_limit`-equivalent + the outer context backstop
   already in `NsjailRunner.Run`.
5. **Verify Finding F is fixed** by porting escape test 20 to run under the
   `GvisorRunner` and asserting the synthesised values (this POC already shows the
   raw result).

Explicitly deferred (out of v1 scope):

- **The other 6 languages** (bash/js/c/cpp/java/verilog) ‚Äî each needs its own rootfs
  contents (compiler toolchains, the JDK, the Icarus ivl base dir). nsjail's
  `resolve*Mounts` already enumerates exactly these file sets, so it is mechanical
  but not free.
- **eBPF tracer compatibility** ‚Äî replaced, not ported (see ¬ß5): a separate piece
  of work to consume gVisor trace points instead of host tracepoints.
- **Full escape-test parity** ‚Äî the 20 escape tests assume nsjail semantics
  (seccomp `SIGSYS` kills, clone/userns behaviour, mount-namespace specifics). Under
  gVisor the *boundary* is the sentry, not host seccomp+namespaces, so the suite
  needs re-derivation, not a 1:1 port. (Several findings the audit raised ‚Äî A
  clone/userns, B io_uring, F /proc ‚Äî are structurally addressed by gVisor
  intercepting all guest syscalls, but that must be re-verified, not assumed.)
- **PIDs limit + the startup-flakiness rough edges** ‚Äî re-test on a real Linux host
  before relying on them.
- **The KVM platform** (performance) ‚Äî only relevant off-WSL2.

### One-line recommendation

gVisor is a viable Phase-7 alternative isolation backend in this environment and is
the clean fix for Finding F. A py3-only `GvisorRunner` proving that end-to-end is a
reasonable next POC step; the bulk of the eventual cost is the per-language rootfs
work and re-deriving the escape suite against the sentry boundary, not getting
gVisor to run.
