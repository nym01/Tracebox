# Tracebox — Security Model

## What Tracebox Is Trying to Protect

Tracebox lets users run code that could be buggy, malicious, or generated by
AI.

The goal is simple: let the code run and produce output, but stop it from
accessing anything outside its own isolated environment.

The code should not be able to:

- Read files from the host machine
- Modify the server
- Access other users' data
- Control Docker
- Affect the operating system

**Think of it like:** giving a stranger a whiteboard in a locked room. They
can write anything on the whiteboard, but they cannot leave the room or touch
anything else in the building.

## The Three Main Security Layers

Tracebox relies on three separate protections.

### 1. Filesystem Isolation

The running program can only see a very small set of files.

It cannot see:
- Host files
- Other users' files
- System secrets
- Most of the container

It only sees:
- The files needed to run
- Its own temporary working folder

**Analogy:** imagine giving someone access to a single hotel room instead of
the entire hotel. They can use what's inside that room, but they can't open
other rooms, access staff areas, or enter the manager's office.

### 2. Syscall Filtering (seccomp)

Programs interact with Linux through system calls. Some of these are
dangerous because they can:

- Change system settings
- Create new privileged environments
- Spy on other processes
- Remount filesystems
- Load kernel modules

Tracebox blocks these dangerous calls. Normal actions still work — reading
files, writing files, creating processes, allocating memory.

**Analogy:** think of a rental car. You can drive it, use the radio, open the
windows — but you cannot replace the engine, rewire the electronics, or
change ownership of the car. The dangerous controls are locked away.

### 3. Resource Limits (cgroups)

Even if code cannot escape, it might try to consume huge amounts of
resources — infinite loops, memory exhaustion, fork bombs (creating
thousands of processes).

Tracebox limits:

- **Memory** — if a program uses too much RAM, it gets killed.
- **Processes** — if a program creates too many child processes, new process
  creation fails.
- **CPU** — if a program tries to use all CPU cores, it gets throttled.

**Analogy:** imagine a child with a prepaid arcade card. They can play games,
but once they use up their credits, they cannot keep spending indefinitely.

## Why All Three Layers Are Needed

No single protection is enough.

- **Filesystem isolation alone** — an attacker might use special system calls
  to bypass it.
- **Syscall filtering alone** — a program could still consume all memory and
  crash the server.
- **Resource limits alone** — a program might still read sensitive files.

**Analogy:** protecting a house requires locks on doors, security cameras,
and an alarm system. One layer alone isn't enough.

## The Real Security Boundary: The Linux Kernel

All three protections are enforced by Linux itself. This means the ultimate
trust point is the Linux kernel — if Linux has a serious vulnerability, an
attacker might bypass all three protections.

**Analogy:** imagine a bank vault. The vault door, alarm system, and cameras
are all excellent — but they all depend on the building's foundation. If the
foundation itself breaks, every security system becomes less effective.

## Things Outside the Sandbox That Still Matter

Even if the sandbox works perfectly, other parts of the system can still be
attacked: the Go server running Tracebox, Docker, the container runtime, and
the Linux kernel itself.

**Important:** the sandbox itself runs with no capabilities — it has almost
no special powers. Even though it appears to run as root inside the sandbox,
it is effectively a "powerless root."

**Analogy:** it's like being given a keycard that says "Administrator" but
opens absolutely nothing.

## Network Access Is Blocked

Programs inside the sandbox cannot access the internet. Testing confirmed no
outbound connections, no routes to the outside world, and only a local
loopback interface.

**Analogy:** the room has no doors, windows, phone, Wi-Fi, or internet
connection. The code can talk to itself, but not to the outside world.

## How Security Could Be Improved Further

**gVisor** adds another layer between the sandbox and the Linux kernel.
Instead of talking directly to the kernel, programs talk to gVisor first.

**Analogy:** instead of speaking directly to the bank manager, every request
goes through a security guard.

**Firecracker** runs each workload inside its own lightweight virtual
machine, with its own kernel.

**Analogy:** instead of giving someone a room in your building, you give them
an entirely separate tiny building. Even if they damage their building, yours
remains safe.

## Known Weaknesses and Limitations

No security system is perfect.

**1. The container uses `--privileged`.** The Docker container has more
permissions than strictly necessary. The sandboxed code itself does not
receive those permissions, but the container's attack surface is larger than
ideal. A better approach would grant only the specific permissions actually
needed.

**2. Security depends on the Linux kernel.** All protections rely on kernel
features. A serious kernel vulnerability could bypass them — this is the
biggest fundamental limitation.

**3. Seccomp uses a deny-list.** Tracebox blocks known dangerous system
calls, but any dangerous syscall that developers forgot to block is allowed
by default. An allow-list would be stricter but harder to maintain across
many programming languages.

**4. No strict disk usage quota.** Individual files are limited in size, but
a program can still create many small files and fill up storage. Current
mitigation: temporary directories are cleaned up and concurrent workloads
are limited.

**5. Some host information leaks through `/proc`.** Programs can still see
information such as CPU model, amount of RAM, kernel version, and system
load. This does not allow escaping the sandbox — it only reveals information
about the machine.

**Analogy:** someone cannot enter your house, but they can look through the
window and see what model of TV you own.

## Bottom Line

Tracebox protects itself using three layers:

- **Filesystem isolation** — code only sees a tiny set of files.
- **Syscall filtering** — dangerous operating system actions are blocked.
- **Resource limits** — memory, CPU, and process abuse are restricted.

Together these create a strong sandbox for running untrusted code.

The biggest remaining risk is not the configuration itself, but the shared
Linux kernel underneath everything. Stronger options like gVisor or
Firecracker would reduce that risk further, but at the cost of additional
complexity and performance overhead.
